# Vibe Coding Navigator Product Strategy

## 1. 한 문장 정의

Vibe Coding Navigator는 개발 지식이 없는 사람이 서비스 아이디어를 AI 개발 도구에 전달 가능한 기획서와 개발 프롬프트로 바꾸도록 돕는 선택형 Wizard입니다.

## 2. 제품 정체성

이 서비스는 개발자를 위한 도구가 아닙니다.

개발을 모르는 사람이 AI 개발자에게 일을 맡길 수 있게 번역해주는 도구입니다.

정의:

> 선택형 진단 도구 + 실시간 기획 문서 생성기 + AI 개발 프롬프트 빌더

## 3. 핵심 사용자

가상의 핵심 사용자는 다음과 같습니다.

- 문과 출신
- 개발 지식 거의 없음
- 서비스 아이디어는 있음
- Cursor, Claude, ChatGPT, Codex 같은 AI 도구가 궁금함
- 하지만 무엇을 어떻게 요청해야 할지 모름
- MVP, API, 아키텍처, 배포 같은 말을 어려워함

이 사용자는 “개발 문서 작성”을 하고 싶은 것이 아닙니다.

자신의 아이디어를 AI 개발 도구가 이해할 수 있는 말로 바꾸고 싶은 것입니다.

## 4. 현재 결론: 온보딩과 용어 번역이 우선

현재 제품은 기능 방향은 맞지만, 초보자에게는 아직 진입 장벽이 있습니다.

다음 개발의 최우선 과제:

1. 첫 화면 온보딩
2. 쉬운 용어 번역
3. 결과물 미리보기

### 4.1 첫 화면이 필요한 이유

바로 Wizard가 시작되면 초보자는 다음 질문을 하게 됩니다.

- 이걸 하면 뭐가 나오지?
- 내가 뭘 준비해야 하지?
- 개발자용 도구인가?
- 끝까지 하면 실제로 뭘 얻지?

첫 화면은 다음 내용을 짧게 설명해야 합니다.

- 아이디어만 있어도 시작할 수 있다.
- 선택지만 골라도 된다.
- 개발 지식이 없어도 된다.
- 끝나면 기획서와 AI 개발 프롬프트를 받는다.
- 약 3~5분이면 첫 초안이 나온다.

추천 첫 문장:

> 아이디어만 있어도 괜찮아요. 몇 가지 선택지를 고르면 AI 개발 도구에 바로 넣을 수 있는 기획서와 개발 프롬프트를 만들어드립니다.

### 4.2 용어 번역이 필요한 이유

초보자는 다음 단어를 이해하기 어렵습니다.

- MVP
- API
- 아키텍처
- 기술 스택
- LocalStorage
- 배포
- 프롬프트
- 환경변수
- Vercel
- Supabase

이 단어를 완전히 없앨 필요는 없습니다. 대신 쉬운 말을 먼저 보여주고 전문용어는 보조로 제공해야 합니다.

용어 번역 예시:

| 전문용어 | 초보자용 표현 |
|---|---|
| MVP | 처음 만들 최소 기능 |
| API | 다른 서비스와 데이터를 주고받는 통로 |
| 아키텍처 | 서비스를 어떤 구조로 만들지 |
| 기술 스택 | 개발에 쓸 도구 조합 |
| 배포 | 인터넷에 공개하기 |
| 프롬프트 | AI 개발자에게 줄 작업 요청서 |
| 환경변수 | 비밀값을 안전하게 보관하는 설정 |
| LocalStorage | 내 브라우저에 임시 저장하기 |

## 5. 단계명 변경 방향

현재 개발자 중심 단계명을 초보자용 표현으로 바꿔야 합니다.

| 기존 단계명 | 초보자용 단계명 |
|---|---|
| 아이디어 입력 | 만들고 싶은 것 고르기 |
| 타겟 · 문제 정의 | 누가 왜 필요할지 정하기 |
| 서비스 형태 선택 | 어떤 모양의 서비스인지 고르기 |
| 레퍼런스 분석 | 참고할 서비스 고르기 |
| MVP 기능 정리 | 처음 만들 기능 고르기 |
| 화면 구조 설계 | 어떤 화면이 필요할지 정하기 |
| 데이터 · API 설계 | 저장할 정보 정하기 |
| 아키텍처 추천 | 만들 방법 정하기 |
| 프롬프트 생성 | AI에게 시킬 말 만들기 |
| 배포 · 운영 체크리스트 | 인터넷에 공개하기 전 확인하기 |

## 6. 참고 서비스 분석

참고 서비스:

- https://apptype.technote.wiki/
- https://apptype.technote.wiki/index-v2

두 서비스의 공통 특징:

- 빈칸 입력보다 선택지 클릭을 먼저 제공한다.
- 질문 수가 제한적이라 부담이 낮다.
- 사용자는 자신의 상황과 가까운 카드를 고른다.
- 답변을 점수화하거나 분류해서 결과를 추천한다.
- 결과 화면에서 앱 유형, 기술 스택, 추천 이유, 다음 행동을 제공한다.
- 회원가입 없이 바로 사용할 수 있다.
- 결과 화면과 체류 시간을 광고 수익화 지점으로 활용한다.

우리 서비스에 적용할 점:

- 모든 단계는 추천 선택지를 먼저 보여준다.
- 선택지를 누르면 입력 초안이 자동 작성된다.
- 사용자는 직접 작성보다 수정과 보완에 집중한다.
- 각 단계 옆에는 현재 단계 프롬프트가 실시간 생성된다.
- 마지막에는 전체 개발 프롬프트와 문서형 결과물이 산출된다.

## 7. 10단계 이후 산출물

10번 단계 이후 사용자는 단순한 텍스트 하나가 아니라 개발 실행 패키지를 받아야 합니다.

필수 산출물:

1. 서비스 기획 요약서
   - 서비스 이름
   - 한 줄 설명
   - 타겟 사용자
   - 해결하려는 문제
   - 핵심 가치
   - 제외할 기능

2. MVP 기능 명세서
   - 반드시 만들 기능
   - 나중에 만들 기능
   - 화면별 기능
   - 사용자 액션
   - 완료 기준

3. 화면 구조 설계서
   - 필요한 화면 목록
   - 각 화면의 역할
   - 주요 UI 컴포넌트
   - 사용자 흐름

4. 기술 설계서
   - 추천 기술 스택
   - 데이터 구조
   - 저장 방식
   - API 필요 여부
   - 배포 방식
   - 환경변수 체크리스트

5. AI 개발 프롬프트
   - 구현 범위
   - 파일 구조
   - 디자인 방향
   - 보안 조건
   - QA 기준
   - README, 커밋, 배포 지시

6. 배포 및 운영 체크리스트
   - 빌드 확인
   - 모바일 화면 확인
   - 환경변수 등록
   - 배포 URL 확인
   - 복사 및 다운로드 동작 확인

결과 화면은 탭 구조가 적합합니다.

- 요약서
- MVP 명세
- 화면 구조
- 기술 설계
- 개발 프롬프트
- 배포 체크리스트

각 탭에는 다음 액션을 둡니다.

- 복사
- Markdown 다운로드
- 새 프로젝트 시작

## 8. ROT(Return on Token) 관점

ROT는 사용자가 쓴 토큰, 시간, 집중력 대비 얼마나 실행 가능한 결과를 얻는지를 뜻합니다.

이 서비스에서 ROT를 높이는 방법:

- 사용자가 직접 긴 글을 쓰지 않게 한다.
- 선택지만 눌러도 문서 초안이 만들어지게 한다.
- 단계별 프롬프트와 전체 프롬프트를 동시에 제공한다.
- 결과물이 바로 AI 개발 도구에 붙여넣을 수 있어야 한다.
- 최종 결과가 개발문서, 명세, 프롬프트, 배포 체크리스트까지 포함해야 한다.

사용자가 느껴야 하는 최종 상태:

> 내 아이디어가 개발 가능한 요구사항 문서와 AI 개발 프롬프트로 바뀌었다.

## 9. 수익화 전략

이 서비스는 무료 진단 도구로 시작하고, 트래픽과 결과 화면 체류 시간을 수익화합니다.

### 9.1 광고 기반 수익화

초기 수익화 모델은 광고가 적합합니다.

추천 광고 위치:

- 10단계 완료 후 결과 화면 상단
- 결과 탭 사이 하단
- 전체 개발 프롬프트 아래
- Markdown 다운로드 버튼 근처

주의할 점:

- 입력 도중에는 광고를 과하게 넣지 않는다.
- 사용자의 사고 흐름을 끊지 않는다.
- 결과 화면처럼 체류 시간이 긴 구간에 배치한다.

환경변수 예시:

```bash
NEXT_PUBLIC_ADSENSE_CLIENT_ID=""
NEXT_PUBLIC_ADSENSE_SLOT_RESULT_TOP=""
NEXT_PUBLIC_ADSENSE_SLOT_RESULT_BOTTOM=""
```

### 9.2 프리미엄 기능

광고만으로는 한계가 있으므로 이후 유료 기능을 고려합니다.

가능한 프리미엄 기능:

- 광고 제거
- 프로젝트 저장
- 결과 문서 PDF 다운로드
- Markdown 다운로드
- Codex/Cursor/Claude용 고급 프롬프트
- 서비스 유형별 템플릿
- AI 자동 분석
- URL/이미지 기반 레퍼런스 분석

### 9.3 제휴 수익

이 서비스와 어울리는 제휴 후보:

- Vercel
- Supabase
- Cursor
- Claude
- OpenAI API
- 도메인 서비스
- 호스팅 서비스
- 노코드/로우코드 툴

## 10. 제품 원칙

- 초보자에게 빈칸부터 보여주지 않는다.
- 선택지를 먼저 주고, 수정할 수 있게 한다.
- 전문용어는 쉬운 말로 번역한다.
- 각 질문마다 왜 묻는지 알려준다.
- 모든 선택은 결과 문서와 프롬프트에 반영되어야 한다.
- 결과는 추천에서 끝나지 않고 실행 가능한 개발 지시문이어야 한다.
- 수익화는 사용 흐름을 방해하지 않고 결과 화면 중심으로 배치한다.
- 무료 사용자는 충분한 가치를 얻고, 고급 사용자는 저장/다운로드/고급 프롬프트에 비용을 지불하게 한다.
